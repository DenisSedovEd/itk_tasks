## Что такое OLTP и OLAP базы данных?

### OLTP:

* **_Назначение_**: Поддержка транзакционной обработки в реальном времени.
* **_Тип операций_**: Частые вставки, обновления, удаления данных.
* **_Характер_** данных: Текущие, оперативные данные.
* **_Пример_**: Системы для обработки заказов, платежей, банковских транзакций.
* **_Особенности_**: Небольшие объемы данных на транзакцию, высокая скорость обработки.

Пример: Интернет-магазин, где клиенты оформляют заказы и оплаты обрабатываются в реальном времени.

### OLAP:

* **_Назначение_**: Анализ больших объемов данных для бизнес-аналитики.
* **_Тип операций_**: Чтение, агрегация и анализ данных.
* **_Характер данных_**: Исторические данные, данные из разных источников.
* **_Пример_**: Системы для отчетности, аналитики продаж, прогнозирования.
* **_Особенности_**: Больших объемы данных, сложные запросы, медленная обработка по сравнению с OLTP.

Пример: BI-платформа для анализа компании за последние несколько лет и прогнозирования на будущее.


## Какие уровни изоляции баз данных вы знаете?

Уровни изоляции баз данных определяют, как транзакции взаимодействуют друг с другом, обеспечивая баланс между производительностью и согласованностью данных. Существует 4 основных уровня изоляции:

1. _Read Uncommited (Чтение неподтвержденных данных)_
* Описание: Транзакция может читать данны, которые еще не были зафиксированы другими транзакциями.
* Проблемы: Возможны "грязные" чтения (dirty reads).
* Пример проблемы: Транзакция может прочитать промежуточные данные, которые затем будут откатаны.

2. _Read Commited (Чтение подтвержденных данных)_
* Описание: Транзакция может читать только те данные, которые уже зафиксированы другими транзакциями.
* Проблемы: Возможно неповторяющиеся чтения (non-repeatable reads)
* Пример проблемы: Данные могут изменить между разными чтениями в одной транзакции.

3. _Repeatable Read (Повторяемое чтение)_
* Описание:Транзакция гарантированно увидит одни и те же данные при повторных чтениях (никакие другие транзакции не могут изменить данные)
* Проблемы: Возможны фантомные чтения (phantom reads).
* Пример проблемы: Новые строки могут быть добавлены в набор данных после первого чтения.

4. _Serializable (Сериализуемый уровень)_
* Описание: Самый строгий уровень изоляции, при котором транзакции выполняются так если бы они шли последовательно, одна за другой.
* Проблемы: Нет, но сильно снижает производительность.
* Пример проблемы: Исключает любые проблемы с параллельностью, но увеличивает задержки выполнения транзакция.

По умолчанию в большинстве случаем используется Read Committed.


## Что такое транзакции?

Транзакции в базах данных - это последовательность одной или нескольких операций (например, вставка, обновления, удаление 
данных), которая выполняется как единое целое. Основная цель транзакции - обеспечить согласованность данных в условиях 
многопользовательской работы и сбоев.

Транзакции подчиняются четырем ключевым свойствам, известным как ACID:
1. Atomicity (атомарность) - транзакция либо выполнятся полностью, либо не выполняется вовсе. Если какая-то операция внутри транзакции не удалась, все изменения откатываются.
2. Consistency (Согласованность) - Транзакция переводит базу данных из одного согласованного состояния в другое. Система гарантирует, что после завершения транзакции все правила целостности данных буду соблюдены.
3. Isolation (Изолированность) - результаты выполнения транзакции не видны другим транзакциям, пока она не завершена. Это помогает избежать ошибок при параллельной работе с базой данных.
4. Durability (Надежность) - После успешного завершения транзакции ее изменения сохраняются в базе данных и остаются даже в случае сбоя системы.

### Пример использования:
Представьте банковскую транзакцию перевода денег. Она может состоять из двух операция:
* Снятие суммы со счета отправителя.
* Пополнение счета получателя.

Обе эти операции должны выполняться в рамках одной транзакции, чтобы не было ситуации, когда деньги сняты, но не зачислены на другой счет.
